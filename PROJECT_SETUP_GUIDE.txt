HOW NEXT.JS ROUTING WORKS (BEGINNER EXPLANATION):
==================================================

File location → URL
├─ app/page.tsx               → http://localhost:3000/
├─ app/about/page.tsx         → http://localhost:3000/about
├─ app/(auth)/sign-in/page.tsx → http://localhost:3000/sign-in
└─ app/products/[id]/page.tsx → http://localhost:3000/products/123

This is called "file-based routing" - the folder structure becomes your URLs!

   In your code:

STEP 4B: PRISMA SCHEMA
======================

File: server/prisma/schema.prisma

This file defines:
1. Database provider (PostgreSQL)
2. All tables (called "models" in Prisma)
3. All columns in each table
4. Relationships between tables

Your schema has these models (tables):

1. TEST MODEL (for testing)
   ```
   model Test {
     id   String @id @default(cuid())  ← Unique ID, auto-generated
     name String                        ← Text field
   }
   ```
   Creates table: test
   Columns: id, name

2. USER MODEL (stores user accounts)
   ```
   model User {
     id            String    @id                ← Unique ID
     name          String                       ← User's name
     email         String                       ← User's email
     emailVerified Boolean   @default(false)   ← Is email verified?
     image         String?                      ← Profile picture (optional)
     createdAt     DateTime  @default(now())   ← When created
     updatedAt     DateTime  @updatedAt        ← Last update time
     sessions      Session[] ← One user has many sessions (relationships)
     accounts      Account[]
     
     @@unique([email])          ← Email must be unique
     @@map("user")              ← Table name in database
   }
   ```

3. SESSION MODEL (stores user sessions/login tokens)
   ```
   model Session {
     id        String
     expiresAt DateTime                  ← When session expires
     token     String                    ← Session token
     createdAt DateTime  @default(now())
     updatedAt DateTime  @updatedAt
     ipAddress String?   ← User's IP address
     userAgent String?   ← User's browser info
     userId    String
     user      User @relation(...)       ← Link to user
     
     @@unique([token])
     @@index([userId])
     @@map("session")
   }
   ```

4. ACCOUNT MODEL (stores OAuth connections like GitHub)
   ```
   model Account {
     id                    String
     accountId             String      ← External account ID (from GitHub)
     providerId            String      ← Provider (github, google, etc)
     userId                String
     user                  User @relation(...)
     accessToken           String?     ← OAuth access token
     refreshToken          String?     ← OAuth refresh token
     accessTokenExpiresAt  DateTime?
     refreshTokenExpiresAt DateTime?
     scope                 String?     ← OAuth scopes (permissions)
     password              String?     ← User password (hashed)
     createdAt             DateTime  @default(now())
     updatedAt             DateTime  @updatedAt
     
     @@index([userId])
     @@map("account")
   }
   ```

5. VERIFICATION MODEL (for email/SMS verification codes)
   ```
   model Verification {
     id         String
     identifier String      ← Email or phone number
     value      String      ← Verification code
     expiresAt  DateTime    ← When code expires
     createdAt  DateTime  @default(now())
     updatedAt  DateTime  @updatedAt
     
     @@index([identifier])
     @@map("verification")
   }
   ```

       // ↑ GitHub OAuth credentials (get from github.com/settings/developers)
     }
   }

STEP 5A: WHAT IS AUTHENTICATION?
==================================

Authentication = "Who are you?"

There are two ways to login:

1. EMAIL & PASSWORD
   - User types email and password
   - System checks if email exists and password matches
   - Creates a session (login token)

2. OAUTH (Social login)
   - User clicks "Login with GitHub"
   - Redirected to GitHub
   - User approves your app
   - GitHub sends back user info
   - You create a session

You're setting up BOTH!

SECTION 16: KEY CONCEPTS FOR BEGINNERS
================================================================================

FRONTEND vs BACKEND:

FRONTEND:
- What user sees (visual interface)
- Runs in browser
- Uses HTML, CSS, JavaScript
- Can't access database directly
- Examples: buttons, forms, pages

BACKEND:
- What user doesn't see (server logic)
- Runs on server/computer
- Handles requests from frontend
- Accesses database
- Can't show visual things
- Examples: login logic, saving data, calculations

REQUESTS & RESPONSES:

REQUEST: Frontend → Backend
  Example: "Frontend asks: What is current user?"
  
RESPONSE: Backend → Frontend
  Example: "Backend responds: Current user is John"

ROUTES:

FRONTEND ROUTES (URLs user sees):
  /              → Homepage
  /sign-in       → Sign in page
  /dashboard     → Dashboard
  /settings      → Settings

BACKEND ROUTES (URLs frontend calls):
  /api/auth/sign-up    → Create account
  /api/auth/sign-in    → Login
  /api/me              → Get current user
  /api/posts           → Get posts

DATABASE:

- Organized tables (like Excel spreadsheets)
- Each table has columns (properties)
- Each row is one record
- Relationships between tables

Example User Table:
| id  | name  | email           | createdAt     |
|-----|-------|-----------------|---------------|
| 1   | John  | john@gmail.com  | 2025-01-01    |
| 2   | Jane  | jane@gmail.com  | 2025-01-02    |

SESSIONS & AUTHENTICATION:

LOGIN FLOW:
1. User enters email/password on frontend
2. Frontend sends to backend
3. Backend checks database
4. If correct, creates SESSION (like a ticket)
5. Frontend stores session token
6. Frontend sends token with future requests
7. Backend verifies token, knows who user is

MIDDLEWARE:

Code that runs before your route handler.

Example:
```
app.use(cors())        ← Runs on EVERY request
app.use(express.json()) ← Runs on EVERY request
app.get('/api/me', ...) ← Runs only on GET /api/me
```

ENVIRONMENT VARIABLES:

Secret settings not committed to GitHub.

Why needed:
- Database passwords
- API keys
- Secrets for signing tokens
- Different values per environment (local, staging, production)

Stored in: .env file
Loaded by: dotenv package
Accessed by: process.env.VARIABLE_NAME
